# 用户探索(Discovery)—— 如何产出证据

**最锋利的一条:你脑子里的"用户想要什么"是假设,不是事实——探索的全部目的,是用最低成本把假设撞碎或撞实,而不是为已有方案找支持。** 大多数访谈失败,是因为 PM 在"推销想法"而非"挖掘真相"。

> 与 `evidence.md` 的分工:**本篇是"如何去拿到证据"(访谈、观察、实验);`evidence.md` 是"拿到后如何推理、分级、引用"。**

## The Mom Test:怎么问才问不出假话

核心:问到的东西要连你妈都没法出于好心而骗你。

- **问过去,不问未来。** "你上次遇到这个是什么时候、当时怎么做的?" 而不是 "你会用这个功能吗?"——对未来意图的回答几乎全是礼貌性谎言。
- **问具体事实和行为,不问观点和赞美。** "你为此花了多少钱/多少时间?"胜过 "你觉得这个点子好吗?"
- **挖凑合方案。** "你现在是怎么解决的?" 已经付出的行动(钱、时间、绕路)才是真需求的证据。
- **别提你的方案。** 一旦说出"我想做个 X",对方就进入捧场模式。先聊他的世界,最后再验证。
- 警惕:赞美、泛泛而谈("我一般都…")、对未来的假设性承诺——这三样都不是证据。

## 机会-解决方案树(把探索结构化)

```
            期望结果(一个可度量的指标)
                      |
        ┌─────────────┼─────────────┐
      机会A          机会B          机会C        ← 用户层面的需求/痛点/欲望
        |
   ┌────┼────┐
 方案1  方案2  方案3                              ← 每个机会下的候选解法
```

- 顶端是**结果**(连回 `metrics.md`),不是功能。
- 中间是从访谈里浮现的**机会**(用户的真实问题),不是你的方案。
- 只有到最底层才谈方案,且每个机会下逼自己想 3 个解法,避免"第一个想法即终点"。
- 价值:让你看见"我们是不是把鸡蛋全压在一个机会上",以及"这个机会到底连不连得回结果"。

## 持续探索,而非一次性调研

- 与其攒一次"大调研",不如每周固定接触 1-2 个真实用户,小步快跑。
- 探索和交付并行:在做当前 spec 的同时,持续为下一个机会积累证据。
- 每次访谈后立刻记录(原话 + 行为 + 出乎意料处),别等记忆衰减。

## 样本与偏差

- **数量**:质性访谈到"听不到新东西"(饱和)通常 5-8 人/细分;但结论只对这个细分成立,别外推。
- **幸存者偏差**:只访问留下来的用户,会系统性错过流失原因——主动找流失/拒绝用户。
- **诱导**:别问"你是不是觉得 X 很麻烦",问"跟我讲讲你上次做 X 的过程"。

## 危险信号

- 访谈里你说话超过对方 → 你在推销,不在探索。
- 拿到的全是"会/想要/觉得好" → 都是未来意图,不算证据。
- 没问凑合方案 → 漏掉了最强的需求信号。
- 只访问了活跃用户 → 幸存者偏差,结论有系统性盲区。
- 把单个用户的原话当成"用户都…" → 见铁律一,不准外推成事实。
