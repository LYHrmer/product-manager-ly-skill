# 自我红队与验收(Critique)—— 质量闸

**最锋利的一条:在交付前,你必须主动攻击自己的产出,直到它扛得住——因为如果你不做,排期会上的工程师、上线后的数据、或竞争对手会替你做,而那时代价高得多。** 这是这个技能的"validate by seeing it":没过验收 rubric,不算完成。

## 通用验收 rubric(任何 PM 产物交付前逐条过)

- [ ] **无捏造**:每个事实/数字/用户原话都有来源标签(铁律一)。没有的标成假设,没有凭空出现的具体数字。
- [ ] **问题先于方案**:有一句话问题陈述,且删掉功能名词后仍成立。
- [ ] **结果先于产出**:成功标准是可度量的用户结果,且配了护栏指标。
- [ ] **有显式不做**:out-of-scope 章节存在且非空。
- [ ] **优先级真排了**:不是一半都 P0;能说出唯一主指标。
- [ ] **开放问题诚实列出**:不确定的没被藏进肯定句。

任一条不过 → 退回去补,不交付。

## 冷读测试(操作化,必做)

1. 假装你是**第一次看到这份文档的资深工程师/设计师**,从头读一遍。
2. 写下他**最可能问倒你的 3 个问题**(通常落在:边界状态、为什么是这个方案而非别的、这个数字哪来的)。
3. 逐个回答。**答不上的,就是还没想清楚的地方**——现在补进文档,而不是等评审会暴露。
4. 补完再读一遍,确认这 3 个问题在文档里已经有答案。

## Pre-mortem(事前验尸)

交付前做一次"假设半年后这事彻底失败了,复盘原因":

- 想象失败已发生,逼自己列出**最可能的 3 个死因**(而不是问"会不会失败")。
- 常见死因:问题其实不存在/不够痛、关键假设错了、指标推不动、被边缘需求拖垮范围、单向门决策草率。
- 对每个死因:现在能做什么把它的概率降下来?改进 spec,或把它写进风险清单。

Pre-mortem 比"你觉得有风险吗"有效得多——它绕过了乐观偏差。

## 给反方做最强论证(Steelman)

- 找出对你方案**最强**的反对意见,并把它论证到比反对者自己更有力。
- 如果你做不出这个最强反方,说明你还没真正理解这个决策——回去补。
- 然后诚实回答:面对这个最强反方,我的方案还成立吗?哪里要改?

## 各产物的额外验收点

- **PRD**:验收标准覆盖了失败/空/加载状态吗?每条需求都连得回目标吗?
- **指标/OKR**:KR 是结果不是任务吗?每个推高指标都有护栏吗?
- **路线图**:每段挂了结果吗?Later 是否被误写成精确承诺?
- **战略**:有没有"放弃什么"?三个"为什么赢"都答得上吗?

## 危险信号(对自己)

- "我觉得挺完整的" → 没做冷读测试,先列出 3 个会被问倒的问题。
- 想不出任何反对意见 → 你在自我说服,做 steelman。
- 风险清单是空的或全是"低" → 乐观偏差,做 pre-mortem。
- 急着交付、跳过这一节 → 这一节就是顶尖和平庸的分水岭,别跳。
