# 干系人与对齐(Stakeholders)

**最锋利的一条:对齐不是让所有人都点头,而是让分歧尽早浮出水面、被听见、然后做出一个大家愿意支持的决定——藏起来的分歧不会消失,它会在最贵的时刻(已经做完时)爆发。** PM 的工作不是讨好,是促成清晰的决策。

## 尽早暴露分歧

- 在**便宜的阶段**(问题界定、目标对齐)主动找出反对意见,而不是等 spec 写完。
- 对"什么不做"的反应,比对"做什么"更能暴露真实诉求(`prioritization.md`)——拿 out-of-scope 清单去探干系人的底。
- 把沉默当风险:没人反对常常意味着没人真正读了/在意,而不是真共识。

## Disagree and Commit(不同意,但承诺执行)

- 充分听取后,有人仍不认同是正常的——目标不是说服每个人,而是:**意见被完整听见 → 决策者拍板 → 所有人(包括反对者)承诺支持执行。**
- 关键前提:反对者要确信自己"被真正听懂了",哪怕结论没采纳他的。所以先复述对方观点到他认可,再表达分歧。
- 决策记录下来:谁拍的板、依据什么、谁反对过什么——便于事后复盘,也防止反复翻烧饼。

## 按受众调整沟通

| 受众 | 他们要什么 | 别给什么 |
|------|------|------|
| 高管 | 结果、战略对齐、取舍和风险 | 实现细节 |
| 工程 | 约束、依赖、为什么、边界 | 含糊的"看着办" |
| 设计 | 问题、用户、目标,而非 UI 指令 | 替他们画好的方案 |
| 客服/销售/运营 | 影响、话术、时间点 | 内部黑话 |

## 叙事备忘录(narrative memo)优于罗列

- 重要决策用**完整句子的叙事**讲清"问题 → 为什么重要 → 选项 → 推荐 → 取舍",而不是一堆 bullet。
- 叙事逼你把逻辑想通——bullet 可以藏住跳跃的推理,完整段落藏不住。
- 一页纸讲清一个决策(见 `templates/one-pager-narrative.md`)。

## 干系人更新(定期)

简短、可扫读:**进展(对照结果指标)→ 风险/阻塞 → 需要的决策/帮助 → 下一步**。重点是"我需要你做什么决定",而不是流水账。

## 危险信号

- 等 spec 写完才找干系人看 → 分歧暴露得太晚、太贵。
- "没人反对"就当成共识 → 可能只是没人读,主动去问。
- 试图说服每个人都同意 → 目标是 commit,不是全体点头。
- 反对者觉得没被听懂 → disagree-and-commit 失效,先复述到他认可。
- 给工程/设计的是 UI 指令而非问题 → 越权且浪费他们的专业。
