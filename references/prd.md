# PRD / 产品规格(Spec)

**最锋利的一条:PRD 是用来对齐和决策的,不是用来留档的——衡量一份 PRD 好坏的唯一标准是,一个没参与讨论的工程师/设计师冷读之后,能不能不来追问就开始干、且干出来的是对的东西。** 写满字数不等于写清楚。

## 结构(每节都要有,且都要短)

1. **问题 & 为什么是现在**——一句话问题陈述(`problem-framing.md`)+ 时机。
2. **目标用户 & 场景**——具体到角色和情境,不是"所有用户"。
3. **成功指标 & 护栏**——可度量的结果 + 反向护栏(`metrics.md`)。先写这个,再写需求。
4. **范围内需求**——编号、平实语言、每条可独立验证。
5. **明确不做(Out-of-scope)**——和需求同等重要(`prioritization.md`)。
6. **验收标准**——Given/When/Then(见下)。
7. **关键假设 & 风险**——来自假设登记(`evidence.md`)。
8. **Open questions**——还没定的,单独列,别藏在正文里。

## 需求怎么写

- **平实语言,编号**。一条需求只说一件事。
- 描述**行为和约束**,不替设计师定 UI、不替工程师定实现(除非那是硬约束)。
  - 坏:"放一个蓝色的右上角分享按钮。"
  - 好:"用户在完成页能把成果分享到至少一个外部渠道;分享动作不超过 2 步。"
- 每条需求问自己:它连回了哪个目标/指标?连不上的,大概率该进 out-of-scope。

## 验收标准:Given / When / Then

```
Given  [前置条件 / 用户处于什么状态]
When   [用户做了什么 / 什么事件发生]
Then   [系统应有的可观察结果]
```

例:
```
Given 用户刚完成一次报修且处于完成页
When  点击"分享给工友"
Then  唤起渠道选择;选定后生成含工单摘要的卡片;失败时给出可重试的明确提示
```

- 覆盖**正常路径 + 边界 + 失败/空/加载状态**——漏掉异常状态是 PRD 最常见的坑。
- 验收标准是"完成"的定义,也是测试用例的来源。

## 写完必做:冷读测试

把 PRD 当成"一个资深工程师第一次看到"来读一遍,列出他**最可能问倒你的 3 个问题**。答不上的,就是还没想清楚的地方——现在补,而不是等到排期会上才暴露。(完整 rubric 见 `critique.md`。)

## 危险信号

- 需求里写死了颜色/按钮位置/技术实现 → 越权替设计/工程做决定,除非是硬约束。
- 没有 out-of-scope 章节 → 范围会蔓延(门 4)。
- 验收标准只有正常路径 → 漏了边界/失败/空/加载状态。
- 成功指标缺失或不可度量 → 门 3 没过,先补。
- Open questions 藏在正文里 → 拎出来单列,否则会被忽略。
- 冒出一个具体数字/用户原话却无来源 → 见铁律一。
