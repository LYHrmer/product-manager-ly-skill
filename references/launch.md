# 上线与发布(Launch)

**最锋利的一条:上线不是终点,是开始度量的起点——"上线了功能 X"是产出;真正的工作是上线后看指标动没动,然后决定迭代、加倍还是砍掉。** 没有"砍掉"这个选项的发布,是把沉没成本当成承诺。

## 上线就绪检查(Launch readiness)

发布前逐项确认,缺一不前:

- **指标埋点先于功能上线**——没有度量就无法判断成败(`metrics.md`)。先确认埋点已就绪、能拿到数。
- **成功 / 失败判据已写定**——上线前就说清"达到什么算成功、什么算该回滚",别事后找补。
- **回滚方案**——怎么快速关掉?有 feature flag 吗?(单向门 vs 双向门,见 `prioritization.md`)
- **边界 / 失败状态**——空状态、错误、加载、降级路径都验过了吗?
- **干系人通知**——客服/销售/运营知道这事、知道怎么答用户吗?(`stakeholders.md`)

## 灰度发布(渐进放量)

- 默认**灰度**而非全量:1% → 10% → 50% → 100%,每档观察护栏指标。
- 用 feature flag 把"发布"和"上线代码"解耦——可随时关,把单向门变成双向门。
- 每档设**叫停条件**:护栏指标恶化超过阈值就停、就回滚,不犹豫。
- 适用 A/B 时,先确认样本量够不够得出结论,别拿噪音当信号(`evidence.md`)。

## GTM(对外)

发布的价值要让目标用户**感知到**:

- 谁需要知道?通过什么渠道?用户为什么要在意(连回价值,不是功能罗列)?
- 内部对齐:客服话术、销售材料、帮助文档是否跟上。
- 别为"发布大"而发布——发布节奏服务于用户价值和指标,不是 PR。

## 上线后:迭代 / 加倍 / 砍掉

发布后用预设判据(不是情绪)做决策:

| 指标表现 | 决策 |
|------|------|
| 主指标动了、护栏没坏 | **加倍**:放量、深化 |
| 主指标没动,但学到了为什么 | **迭代**:改一版再试 |
| 主指标没动,且看不到改的方向 | **砍掉/回滚**:止损,把资源还给更高优先级 |

诚实执行"砍掉"是顶尖 PM 的标志——它需要对抗沉没成本和面子。把它写进发布计划,让砍掉成为合法选项。

## 危险信号

- 埋点还没好就上线 → 上线即"盲飞",无法判断成败。
- 没定回滚方案/叫停条件 → 出事只能干瞪眼。
- 默认全量发布 → 放弃了灰度这道最便宜的保险。
- "都做到这儿了不能砍" → 沉没成本谬误,回到预设判据。
- 客服/销售不知情 → 用户问起来无人能答,体验崩在最后一公里。
